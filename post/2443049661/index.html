MySQL数据库如何调优及案例分析？

分类：MySQL， 作者： 徐梦旗， 发布于： 2024-03-08 20:00， 字数：1.7k，预计阅读：7分钟
#总结 #图解

1. 数据库调优

1.1. 为什么需要数据库调优？

数据库调优是本质上为了提供更好的服务，体现在以下几个指标：

  • 响应时间（Response Time，RT）
  • 每秒事务处理量（Transaction Per Second，TPS）
  • 每秒查询处理量（Query Per Second，QPS）

1.3. 数据库调优的手段有哪些？

数据库调优有以下几种手段：

  • 数据库层面：
    • 修改数据库的配置，如设置合适的连接数、提高缓冲区的大小、开启慢查询日志等。
    • 采用高可用方案、提升硬件配置。
  • 表层面：
    • 为表字段选择合适的类型，创建合适的索引。
    • 适当冗余字段，减少连接查询。
  • SQL语句层面：
    • 查看SQL语句的执行计划，优化SQL语句，避免全表扫描和索引失效。
    • 优化SQL语句，如LIMIT优化、子查询优化、小表驱动大表等。

1.4. 数据库出现性能问题时如何排查？

  1. 开启慢查询日志，定位查询慢的SQL语句；
  2. 使用EXPLAIN命令看SQL语句的执行计划；
  3. 优化SQL语句或者创建合适的索引。

2. 执行计划

2.1. 如何查看SQL语句的执行计划？

可以通过EXPLAIN命令来查看SQL语句的执行计划，输出结果中包含以下几个要素：

  • id：查询标识号
  • select_type：查询类型
  • table：表
  • partitions：分区信息
  • type：连接类型
  • possible_keys：可能用到到索引
  • key：使用的索引
  • key_len：使用的索引的长度
  • ref：关联的索引列
  • rows：预估扫描行数
  • filtered：过滤百分比
  • Extra：额外信息

2.2. select_type

  • 普通查询
    • SIMPLE
    • PRIMARY
  • 子查询
    • SUBQUERY
    • DEPENDENT SUBQUERY
  • 联合查询
    • UNION
    • UNION RESULT
    • DEPENDENT UNION
  • 其它
    • DERIVED
-- SIMPLE 简单查询（不存在子查询或联合查询）
explain select * from user where id = 1;

-- PRIMARY, UNION, UNION RESULT 联合查询
explain select * from user a where id = 1 union select * from user b where id = 2;

-- PRIMARY, DERIVED, UNION, UNION RESULT 带有派生表的联合查询
explain select * from (select * from user a where id = 1 union select * from user b where id = 2) c;

-- PRIMARY, DEPENDENT SUBQUERY, DEPENDENT UNION, UNION RESULT 依赖子查询和依赖联合查询
explain select * from user a where id in (select id from user b where balance > 100 union select id from user c where balance < 200);

-- PRIMARY, SUBQUERY 子查询
explain select a.id, (select max(balance) as max_balance from user) from user a;

-- PRIMARY, DEPENDENT SUBQUERY 依赖子查询
explain select a.id, (select min(b.balance) as max_balance from user b where b.balance > a.balance) from user a;

2.3. type

  • system
  • const
  • eq_ref
  • ref
  • ref_or_null
  • range
  • index_merge
  • index
  • all
-- system

-- const 唯一索引或主键等值查询，常量级别；在索引树上找到第一个满足条件的便停止查找
explain select * from user where id = 23;

-- eq_ref 唯一索引或主键作为连接查询的条件
explain select a.* from user a, user b where a.id = b.id;

-- ref 非唯一索引等值查询；在索引树上找到第一个不满住条件的便停止查找
explain select * from user where job = '工程师';

-- ref_or_null 非唯一索引等值查询且允许列为空
explain select * from user where job = '工程师' or job is null;

-- range 范围查询；范围扫描区间
explain select * from user where id > 80;
explain select * from user where id in (3, 4);
explain select * from user where job like '工%';

-- index_merge 索引合并，同时使用两个索引
explain select * from user where id > 10 or job = '工程师';

-- index 扫描整个索引树
explain select job from user;

-- ALL 全表扫描
explain select * from user where email = 'test@google.com';

2.4. Extra

  • Using filesort：使用文件排序
  • Using where：在服务层进行过滤
  • Using index：使用覆盖索引
  • Using index condition：使用索引条件下推
  • Using union：使用索引合并
-- Using filesort 使用文件排序
explain select * from user where id > 10 order by id_no desc limit 0, 10;

-- Using index 使用覆盖索引
explain select id_no from user where id_no = '123';

-- Using where 使用服务层过滤
explain select * from user where id > 10 and balance > 10;

-- Using union 使用索引合并
explain select * from user where id > 10 or job = '工程师';

-- Using index condition 使用索引条件下推，减少回表的次数
explain select * from user where country like '朝鲜%' and city = '贵州省';

3. 主从复制

3.1. 为什么需要主从复制？

  • 用来备份主库，保证数据的安全。
  • 支撑读写分离，提高性能。
  • 支撑故障转移，提高可用性。

3.2. 主从复制的原理是什么？

由主库的IO线程生成binlog，由从库的IO线程生成中继日志relay log，而后SQL线程重放SQL。

3.3. 主从复制的策略有哪些？

  • 基于Pos主从复制
  • 基于GTID主从复制
  • 一主多从，一主多级从，互为主从
  • 半同步复制机制

3.4. 如何解决主从复制延迟的问题？

主从复制延迟瓶颈在于磁盘IO和网络IO。

  • 修改从数据库的刷盘策略提高性能，降低安全性
  • 多个从数据库，分散从库的读压力
  • 从库使用多线程

4. 读写分离

4.1. 为什么需要读写分离？

提高读的能力，分摊读压力

4.2. 实现读写分离的工具有哪些？

  • atlas
  • MyCat
  • MySQL Proxy
  • MySQL Router
  • Sharding Sphere

5. 分库分表

5.1. 为何需要分库分表？

突破MySQL单库单表的瓶颈，提高数据库的性能

5.2. 分库分表的策略有哪些？

切分与聚合。

  • 垂直切分
    • 垂直分库，按业务分库
    • 垂直分表，大表拆小表
  • 水平切分
    • 水平分库
    • 水平分表

5.3. 数据切片有哪些方法？

  • 取模分片
  • 按范围分片
  • 按时间分片
  • 按枚举值分片
  • 自定义分片

5.4. 分库分表的缺点有哪些？

  • 分库，业务间需要通过接口交互
  • 整加事务管理难度

5.5. 实现分库分表的工具有哪些？

  • Sharding Sphere
  • MyCat

6. 高可用集群

6.1. 为什么需要高可用集群？

完成自动的故障转移，提高系统的可用性

6.2. 高可用集群方案有哪些问题？

  • 主键冲突
  • 热备节点
  • 检查机制完善
  • 数据一致性保证
