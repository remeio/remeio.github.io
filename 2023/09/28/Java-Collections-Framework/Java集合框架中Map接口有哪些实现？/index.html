Java集合框架中Map接口有哪些实现？

1. Map接口详解

1.1. Map接口有哪些操作？

Map是指将键映射到对应的值。一个Map不能包含重复的键：每个键最多只能映射一个值。Map接口包括以下操作：

  • 基础操作（Basic operations）
    如put，get，remove，containsKey，containsValue，size以及empty方法。

  • 批量操作（Bulk operations）
    如putAll，clear方法。

  • 集合视图（Collection views）
    如keySet，entrySet，values方法。

2. HashMap详解

2.1. HashMap底层是怎么实现的？

  • 从底层数据结构上看，HashMap使用了数组，链表以及红黑树；
  • 从存储元素上看，HashMap允许存储null键和null值，是无序的；
  • 从线程安全上看，HashMap是线程不安全的，采用fast-fail机制。

2.2. HashMap是如何解决哈希冲突的？

哈希冲突是指两个对象经过哈希函数计算得到的哈希值相同。解决哈希冲突的方法有：

  • 开放定址法：冲突时寻找新的空闲的内存地址。
  • 再哈希法：再次计算哈希值直到不发生哈希冲突。
  • 链表法：利用链表维护哈希冲突的节点。
  • 建立公共溢出区：额外开辟一块内存区域来维护冲突的节点。

HashMap使用链表法解决哈希冲突，哈希值相同的键所对应的节点由一条链表维护。

在一定的条件下（最小树化容量为64，树化阈值为8），HashMap会将链表转为红黑树，原因是当链表中的节点越来越多时，查询的时间复杂度会由O(1)退化为O(n)；而使用红黑树时，查询的时间复杂度为O(logn)，比链表的查询效率要高，代价是新增和修改操作需要维护红黑树的结构，故在一定条件下，红黑树也会转为链表（链化阈值为6）。

2.3. HashMap是如何添加键值对的？

1. 计算哈希值及下标，并找到头节点
通过哈希函数计算键的哈希值，并通过数组长度同哈希值的位运算得到数组下标。如果数组中该下标不存在节点，则在该下标创建键值对的节点。
2. 头节点判断
如果数组中该下标存在节点，且该节点的哈希值和键同要添加的键的相同，则修改该节点对应的值，否者需要判断该节点是树节点还是链表节点。
3. 如果头节点是红黑树，则在红黑树中添加或修改节点
如果是树节点，代表使用的是红黑树解决哈希冲突，则在红黑树中找到对应的位置创建键值对的节点或修改节点对应的值。
4. 如果头节点是链表，则在链表中添加或修改节点
如果是链表节点，代表使用的是链表解决哈希冲突，则在链表中找到对应的位置创建键值对的节点或修改节点对应的值。在一定的条件下，链表会转化为红黑树。
5. 超出阈值进行扩容
最后，如果键值对的个数大于扩容的阈值，即数组长度乘以负载因子，会进行扩容。

public V put(K key, V value) {
	return putVal(hash(key), key, value, false, true);
}

// 通过哈希函数计算键的哈希值
static final int hash(Object key) {
	int h;
	return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}

final V putVal(int hash, K key, V value, boolean onlyIfAbsent,
                   boolean evict) {
	Node<K,V>[] tab; Node<K,V> p; int n, i;
	// 第一次添加元素需要扩容
	if ((tab = table) == null || (n = tab.length) == 0)
		n = (tab = resize()).length;
	// 通过数组大小同哈希值的位运算得到数组下标
	if ((p = tab[i = (n - 1) & hash]) == null)
		// 如果数组中该下标不存在节点，则在该下标创建键值对的节点
		tab[i] = newNode(hash, key, value, null);
	else {
		Node<K,V> e; K k;
		// 如果数组中该下标存在节点，且该节点的哈希值和键同要添加的键的相同，则修改该节点对应的值
		if (p.hash == hash &&
			((k = p.key) == key || (key != null && key.equals(k))))
			e = p;
		// 判断该节点是树节点还是链表节点
		else if (p instanceof TreeNode)
			// 如果是树节点，代表使用的是红黑树解决哈希冲突，则在红黑树中找到对应的位置创建键值对的节点或修改节点对应的值
			e = ((TreeNode<K,V>)p).putTreeVal(this, tab, hash, key, value);
		else {
			// 如果是链表节点，代表使用的是链表解决哈希冲突，则在链表中找到对应的位置创建键值对的节点或修改节点对应的值
			for (int binCount = 0; ; ++binCount) {
				if ((e = p.next) == null) {
					p.next = newNode(hash, key, value, null);
					if (binCount >= TREEIFY_THRESHOLD - 1) // -1 for 1st
						// 在一定的条件下，链表会转化为红黑树
						treeifyBin(tab, hash);
					break;
				}
				if (e.hash == hash &&
					((k = e.key) == key || (key != null && key.equals(k))))
					break;
				p = e;
			}
		}
		if (e != null) { // existing mapping for key
			// 修改节点对应的值
			V oldValue = e.value;
			if (!onlyIfAbsent || oldValue == null)
				e.value = value;
			afterNodeAccess(e);
			return oldValue;
		}
	}
	++modCount;
	// 如果键值对的个数大于扩容的阈值，即容量乘以负载因子，会进行扩容
	if (++size > threshold)
		resize();
	afterNodeInsertion(evict);
	return null;
}

2.4. HashMap为什么需要扩容？什么是负载因子？

如果HashMap不进行扩容，当存储的键值对的个数较多时，由于数组空闲位置较少，会频繁发生哈希冲突，导致大量的键值对存放在链表或红黑树中使得查询速度降低。

当键值对的个数大于扩容的阈值（即数组长度乘以负载因子）时，HashMap会进行扩容，数组长度扩容为原数组的2倍。负载因子越大，哈希冲突的几率越小，浪费的内存空间越多；负载因子越小，哈希冲突的几率越大，浪费的内存空间越少。负载因子默认为0.75。

2.5. HashMap是如何根据键找到值的？

1. 计算哈希值及下标，并找到头节点
通过哈希函数计算键的哈希值，并通过数组长度同哈希值的位运算得到数组下标。
2. 头节点判断
如果数组中该下标存在节点，且该节点的哈希值和键同要添加的键的相同，则返回该节点对应的值，否者需要判断该节点是树节点还是链表节点。
3. 如果头节点是红黑树，则在红黑树中查找键对应的值
如果是树节点，则遍历红黑树查找键对应的节点，并返回该节点对应的值，时间复杂度为O(logn)。
3. 如果头节点是链表，则在红黑树中查找键对应的值
如果是链表节点，则遍历链表查找键对应的节点，并返回该节点对应的值，时间复杂度为O(n)。

public V get(Object key) {
	Node<K,V> e;
	return (e = getNode(hash(key), key)) == null ? null : e.value;
}

final Node<K,V> getNode(int hash, Object key) {
	Node<K,V>[] tab; Node<K,V> first, e; int n; K k;
	if ((tab = table) != null && (n = tab.length) > 0 &&
		(first = tab[(n - 1) & hash]) != null) {
		// 如果数组中该下标存在节点，且该节点的哈希值和键同要添加的键的相同，则返回该节点对应的值
		if (first.hash == hash && // always check first node
			((k = first.key) == key || (key != null && key.equals(k))))
			return first;
		if ((e = first.next) != null) {
			// 如果是树节点，则遍历红黑树查找键对应的节点，并返回该节点对应的值
			if (first instanceof TreeNode)
				return ((TreeNode<K,V>)first).getTreeNode(hash, key);
			// 如果是链表节点，则遍历链表查找键对应的节点，并返回该节点对应的值
			do {
				if (e.hash == hash &&
					((k = e.key) == key || (key != null && key.equals(k))))
					return e;
			} while ((e = e.next) != null);
		}
	}
	return null;
}

3. LinkedHashMap详解

3.1. LinkedHashMap底层是怎么实现的？

  • 从继承关系上看，LinkedHashMap继承了HashMap，实现了Map接口；
  • 从底层数据结构上看，相比于HashMap，LinkedHashMap额外添加了一个双向链表，用于维护顺序；
  • 从存储元素上看，LinkedHashMap允许存储null键和null值，是有序的；
  • 从线程安全上看，LinkedHashMap是线程不安全的，采用fast-fail机制。

3.2. LinkedHashMap是如何实现有序的？

LinkedHashMap是在HashMap的基础上，添加了双向链表来维护顺序，支持插入顺序和访问顺序，默认为插入顺序。底层原理为LinkedHashMap使用了LinkedHashMap#Entry节点来代替了HashMap#Node节点，相比于Node节点，Entry节点多了before指针和after指针，即维护了一个双向链表。当创建新节点时，LinkedHashMap会将新节点添加到双向链表的尾部。

Node<K,V> newNode(int hash, K key, V value, Node<K,V> e) {
	LinkedHashMap.Entry<K,V> p =
		new LinkedHashMap.Entry<K,V>(hash, key, value, e);
	linkNodeLast(p);
	return p;
}

// link at the end of list
private void linkNodeLast(LinkedHashMap.Entry<K,V> p) {
	LinkedHashMap.Entry<K,V> last = tail;
	tail = p;
	if (last == null)
		head = p;
	else {
		p.before = last;
		last.after = p;
	}
}

当访问节点且使用的是访问顺序时，LinkHashMap会将被访问的节点移到双向链表的尾部。

void afterNodeAccess(Node<K,V> e) { // move node to last
	LinkedHashMap.Entry<K,V> last;
	if (accessOrder && (last = tail) != e) {
		LinkedHashMap.Entry<K,V> p =
			(LinkedHashMap.Entry<K,V>)e, b = p.before, a = p.after;
		p.after = null;
		if (b == null)
			head = a;
		else
			b.after = a;
		if (a != null)
			a.before = b;
		else
			last = b;
		if (last == null)
			head = p;
		else {
			p.before = last;
			last.after = p;
		}
		tail = p;
		++modCount;
	}
}

3.3. LinkedHashMap和HashMap有什么区别？

LinkedHashMap可以理解为有顺序的HashMap，额外添加了一个双向链表，用于维护顺序。
